## 一、问题核实结果

### 高优先级问题

#### 1. ✅ 已确认：`postgres.go` 单文件过大（1126 行）
**报告位置**: 8.2 改进建议 - 高优先级  
**核实结果**: **确认存在** - 文件实际 1126 行，包含所有 Store 操作  
**影响**: 可维护性差，代码审查困难  
**建议**: 按实体拆分为多个文件

#### 2. ✅ 已确认：审计日志 `Close()` 不等待 worker 完成
**报告位置**: 5.2 安全关注点 #3, 8.2 改进建议 - 高优先级  
**核实结果**: **确认存在** - `audit.go:76-78` 仅关闭 `stopChan`，未使用 WaitGroup  
**代码证据**:
```go
func (s *AuditService) Close() {
    close(s.stopChan)  // 仅关闭信号，worker 可能还在处理
}
```
**影响**: 服务关闭时可能丢失审计日志  
**建议**: 使用 WaitGroup 确保 worker 完成

#### 3. ✅ 已确认：自定义 Metrics 实现非标准
**报告位置**: 7.2 监控, 8.2 改进建议 - 高优先级  
**核实结果**: **确认存在** - 自定义实现（218 行），Histogram 仅存储单值无桶分布  
**代码证据**: `metrics.go:158-176` 输出格式为 `name value`，无桶信息  
**影响**: 与 Prometheus 生态不兼容，无法使用百分位等高级功能  
**建议**: 使用官方 `prometheus/client_golang`

### 中优先级问题

#### 4. ✅ 已确认：Redis `Keys()` 命令阻塞
**报告位置**: 3.1 缓存层, 8.2 改进建议 - 中优先级  
**核实结果**: **确认存在** - `redis.go:368-378` 使用 `client.Keys(ctx, pattern)`  
**代码证据**:
```go
func (c *RedisCache) DeletePattern(ctx context.Context, pattern string) error {
    keys, err := c.client.Keys(ctx, pattern).Result()  // 阻塞命令
    // ...
}
```
**影响**: 大数据集上阻塞 Redis，影响其他请求  
**建议**: 改用 `SCAN` 命令

#### 5. ✅ 已确认：分布式限流缺失
**报告位置**: 5.2 安全关注点 #1, 8.2 改进建议 - 中优先级  
**核实结果**: **确认存在** - `ratelimit.go` 使用内存 map 存储客户端信息  
**代码证据**: `ratelimit.go:18-25` - `clients map[string]*clientInfo`  
**影响**: 多实例部署时各实例独立计数，限流失效  
**建议**: 使用 Redis 实现分布式限流

#### 6. ⚠️ 部分确认：邮件模板内联字符串
**报告位置**: 3.6 Service 层 - email.go, 8.2 改进建议 - 中优先级  
**核实结果**: **已使用 html/template，但模板为内联字符串**  
**代码证据**: `email.go:204-236` 和 `email.go:261-293` - 模板为字符串变量  
**实际情况**: 已使用 `html/template` 渲染，但模板未外部化  
**影响**: 修改模板需重新编译，维护不便  
**建议**: 将模板提取到 `templates/email/` 目录

#### 7. ✅ 已确认：TOTP 无恢复码
**报告位置**: 5.2 安全关注点 #4, 8.2 改进建议 - 中优先级  
**核实结果**: **确认缺失** - `mfa.go` 仅实现 TOTP，无恢复码功能  
**代码证据**: 用户模型无 `recovery_codes` 字段，无相关方法  
**影响**: 用户设备丢失后无法恢复 MFA  
**建议**: 添加备用恢复码功能

#### 8. ✅ 已确认：社交登录提供商硬编码
**报告位置**: 5.2 安全关注点 #5, 8.2 改进建议 - 中优先级  
**核实结果**: **确认存在** - `social.go:88-109` 硬编码 Google 和 GitHub  
**代码证据**:
```go
if googleClientID != "" {
    providers["google"] = &OAuthProvider{...}  // 硬编码
}
if githubClientID != "" {
    providers["github"] = &OAuthProvider{...}  // 硬编码
}
```
**影响**: 添加新提供商需修改核心代码（违反开闭原则）  
**建议**: 使用插件/注册模式

### 低优先级问题

#### 9. ⚠️ 部分确认：Handler 文件过小
**报告位置**: 3.5 Handler 层, 8.2 改进建议 - 低优先级  
**核实结果**: **部分确认** - login.go (54行) 和 register.go (53行) 确实较小  
**实际情况**: 文件较小但职责清晰，合并不一定带来明显收益  
**影响**: 轻微的文件碎片化  
**建议**: 可考虑合并，但优先级较低

#### 10. ✅ 已确认：内存缓存无大小限制
**报告位置**: 3.1 缓存层, 8.2 改进建议 - 低优先级  
**核实结果**: **确认存在** - `MemoryCache` 使用 `map[string]cacheItem`，无上限  
**代码证据**: `redis.go:100-107` - 仅有过期清理，无数量限制  
**影响**: 理论上可能无限增长（实际受 TTL 限制）  
**建议**: 添加 LRU 淘汰策略

#### 11. ⚠️ 部分确认：两套错误码体系
**报告位置**: 3.4 错误系统, 8.2 改进建议 - 低优先级  
**核实结果**: **存在但已统一管理**  
**实际情况**: 
- `internal/errors/errors.go` 定义 `ErrorCode` 常量（如 `ErrCodeInvalidCredentials`）
- `handler/helpers.go` 使用相同的 `apperrors.ErrorCode` 类型
- `validationErrors` 映射表将 validator 错误映射到错误码
- 不存在完全独立的两套体系，但有 Handler 特定的错误消息常量

**影响**: 轻微的概念混淆，实际使用统一  
**建议**: 保持现状，文档说明即可

**目标**: 使用 Redis 实现跨实例的分布式限流

2. 实现 `RedisRateLimiter` 结构体：
   - 使用 Redis `INCR` + `EXPIRE` 实现滑动窗口计数
   - 或使用 Lua 脚本实现原子操作（推荐，减少网络往返）
3. 保留现有 `RateLimiter` 作为降级方案（内存限流）

**Lua 脚本示例**:
```lua
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local current = redis.call('INCR', key)
if current == 1 then
    redis.call('EXPIRE', key, window)
end
if current > limit then
    return 0
end
return 1
```

**验收标准**:
- 多实例共享限流计数（通过 Redis）
- Redis 不可用时降级到内存限流
