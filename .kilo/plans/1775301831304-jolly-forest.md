# SSO服务性能优化执行计划

## 优化目标

基于压测结果，系统性优化SSO服务性能：
- 登录/注册RPS：从~11提升至15-20（BCRYPT_COST=12）
- UserInfo RPS：从150提升至500+
- 整体响应时间：降低30-50%

## 阶段1：快速优化（1-2天）

### 1.1 消除登录失败冗余DB查询

**问题**：`handleLoginFailure`在密码错误时再次调用`GetByEmail`查询用户，但`validateUserCredentials`已经查询过一次。

**文件**：`internal/service/auth.go`

**改动**：
- 修改`handleLoginFailure`签名，接收`*model.User`参数
- 移除内部的`GetByEmail`调用
- 更新调用处传递user对象

**预期收益**：登录失败场景减少1次DB查询（~5-10ms）

### 1.2 异步发送验证邮件

**问题**：注册时串行发送验证邮件，阻塞响应。

**文件**：`internal/service/auth.go:191-195`

**改动**：
- 使用goroutine异步发送验证邮件
- 使用`context.Background()`避免context取消影响邮件发送
- 保留错误日志

**预期收益**：注册响应时间减少50-200ms

### 1.3 并行化登录成功操作

**问题**：`handleLoginSuccess`中`ResetLoginAttempts`、审计日志、Token生成串行执行。

**文件**：`internal/service/auth.go:372-392`

**改动**：
- 使用`sync.WaitGroup`并行执行`ResetLoginAttempts`和审计日志
- Token生成必须等待（依赖用户信息）
- 错误处理保持非阻塞

**预期收益**：登录成功场景响应时间减少10-30ms

### 1.4 数据库索引验证

**问题**：确保关键查询有索引支持。

**文件**：`migrations/`

**改动**：
- 检查现有索引（006、007、008迁移文件）
- 如缺失则创建新迁移添加索引：
  - `idx_users_email` ON `users(email)`
  - `idx_tokens_access_token` ON `tokens(access_token)`
  - `idx_tokens_refresh_token` ON `tokens(refresh_token)`

**预期收益**：查询延迟降低50%

### 1.5 连接池配置优化

**问题**：当前默认连接池配置可能不适合高并发场景。

**文件**：`internal/config/config.go`、`.env.example`

**改动**：
- 调整默认值：`DBMaxOpenConns=100`、`DBMaxIdleConns=50`
- 添加`DBConnMaxIdleTime`配置（1分钟）
- 更新`.env.example`注释说明

**预期收益**：高并发下连接等待时间减少

## 阶段2：缓存优化（3-5天）

### 2.1 UserInfo接口添加缓存

**问题**：UserInfoHandler每次都查询数据库。

**文件**：`internal/handler/userinfo.go`

**改动**：
- 在`UserInfoHandler`中添加cache字段
- 查询前先尝试从缓存获取
- 缓存未命中则查询DB并写入缓存（5分钟TTL）
- 用户信息变更时失效缓存

**预期收益**：UserInfo RPS从150提升至500+

### 2.2 修复Auth Middleware使用Request Context

**问题**：Token黑名单检查使用`context.Background()`，不尊重请求取消。

**文件**：`internal/middleware/auth.go`

**改动**：
- 将`context.Background()`替换为`r.Context()`
- 确保cache/DB查询尊重请求超时

**预期收益**：取消的请求不再浪费DB/Redis资源

### 2.3 优化MemoryCache.DeletePattern性能

**问题**：`DeletePattern`在写锁下遍历所有key，O(n)阻塞。

**文件**：`internal/cache/redis.go`

**改动**：
- 使用RLock遍历收集需要删除的key
- 释放RLock后逐个获取Lock删除
- 或使用SCAN命令批量删除

**预期收益**：缓存清理操作不阻塞其他读写

### 2.4 添加缓存穿透防护

**问题**：多个并发请求同时请求同一不存在的key，全部打到DB。

**文件**：`internal/cache/redis.go`

**改动**：
- 使用`singleflight`包防止缓存击穿
- 同一key同时只有一个请求查询DB
- 其他请求等待结果

**预期收益**：高并发下DB压力显著降低

## 阶段3：高级优化（1-2周）

### 3.1 MFA Recovery Code O(n) Bcrypt优化

**问题**：`VerifyAndUseMFARecoveryCode`加载所有未使用recovery code并逐个bcrypt比较。

**文件**：`internal/store/postgres/mfa_recovery.go`

**改动**：
- 存储recovery code的哈希值而非明文
- 查询时计算输入code的哈希并进行DB匹配
- 或使用更快的比较方法（如HMAC）

**预期收益**：MFA验证延迟从O(n×bcrypt)降至O(1)

### 3.2 Rate Limiter全局锁优化

**问题**：内存限流器使用单一mutex，高并发下成为瓶颈。

**文件**：`internal/middleware/ratelimit.go`

**改动**：
- 使用分片锁（sharded locks）
- 根据IP hash分配到不同分片
- 或使用`sync.Map`减少锁竞争

**预期收益**：限流器在高并发下不再成为瓶颈

### 3.3 Redis重连逻辑

**问题**：Redis断开后无重连逻辑。

**文件**：`internal/cache/redis.go`

**改动**：
- 添加连接健康检查
- 断开后自动重连
- 重连失败降级到MemoryCache

**预期收益**：Redis故障恢复能力增强

### 3.4 异步审计日志

**问题**：审计日志可能阻塞主流程。

**文件**：`internal/service/auth.go`

**改动**：
- 使用channel缓冲审计日志
- 后台goroutine消费
- channel满时丢弃事件（不阻塞主流程）

**预期收益**：审计日志不再影响主流程性能

## 验证计划

每个阶段完成后执行：
1. `make test` - 确保所有测试通过
2. `make lint` - 代码质量检查
3. 运行对应压测场景验证性能提升
4. 更新压测报告

## 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| 异步操作导致数据不一致 | 保留错误日志，关键操作仍同步 |
| 缓存不一致 | 用户信息变更时失效缓存 |
| 并发引入竞态条件 | 使用race detector测试 |
| 优化后性能未达预期 | 每阶段独立验证，可回滚 |
