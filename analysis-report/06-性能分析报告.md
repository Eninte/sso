## 一、性能架构概述

### 1.1 技术栈性能特性

| 组件 | 版本 | 性能特性 |
|------|------|----------|
| Go | 1.24.2 | 高性能、并发友好 |
| gorilla/mux | v1.8.1 | 路由性能中等 |
| PostgreSQL | 15 | ACID、高并发 |
| Redis | 7 | 高性能缓存 |
| bcrypt | - | 密码哈希（故意慢） |
| RSA-2048 | - | JWT签名（CPU密集） |

### 1.2 潜在瓶颈

```
请求处理流程:
Client → HTTP Server → Middleware → Handler → Service → Store → PostgreSQL
                                          ↓
                                       JWT签名/验证
                                          ↓
                                       bcrypt哈希/验证
```

**关键路径**:
1. JWT 签名/验证 - CPU密集
2. bcrypt 哈希/验证 - CPU密集（设计为慢）
3. 数据库查询 - I/O密集
4. 限流器 - 内存/并发

## 五、限流性能

### 5.1 当前实现

```go
// internal/middleware/ratelimit.go
type RateLimiter struct {
    mu       sync.Mutex
    visitors map[string]*visitor
}
```

**问题**:
- ❌ 单机限流，不支持分布式
- ❌ 内存存储，重启丢失
- ❌ 无持久化
- ❌ Mutex 可能成为瓶颈

### 5.2 性能瓶颈

**Mutex 竞争**:
```go
func (rl *RateLimiter) isAllowed(key string) bool {
    rl.mu.Lock()  // 全局锁
    defer rl.mu.Unlock()
    // ...
}
```

**问题**: 高并发时 Mutex 竞争严重

**建议**: 使用分片锁或 Redis

### 5.3 分布式限流方案

**Redis 实现**:
```go
// 使用 Redis INCR + EXPIRE
func (rl *RedisRateLimiter) isAllowed(ctx context.Context, key string) bool {
    count, err := rl.redis.Incr(ctx, key).Result()
    if err != nil {
        return false
    }
    if count == 1 {
        rl.redis.Expire(ctx, key, rl.window)
    }
    return count <= rl.limit
}
```

## 六、并发安全分析

### 6.1 潜在竞态条件

| 位置 | 问题 | 风险 |
|------|------|------|
| auth.go:76-82 | TOCTOU竞态 | 高 |
| auth.go:139 | 时间比较竞态 | 中 |
| oauth.go:148 | 授权码检查竞态 | 高 |
| ratelimit.go | 全局锁竞争 | 中 |

### 6.2 数据库事务

**问题**: 关键操作未使用事务

```go
// auth.go - 登录成功后
s.store.UpdateLoginAttempts(ctx, user.ID, 0, time.Time{})  // 操作1
s.store.Update(ctx, user)                                   // 操作2
// 两个操作非原子性
```

**建议**: 使用事务确保原子性

```go
tx, err := s.store.BeginTx(ctx)
if err != nil {
    return err
}
defer tx.Rollback()

if err := tx.UpdateLoginAttempts(ctx, user.ID, 0, time.Time{}); err != nil {
    return err
}
if err := tx.Update(ctx, user); err != nil {
    return err
}
return tx.Commit()
```

### 6.3 并发访问模式

**安全的并发访问**:
- ✅ JWT Service 无状态
- ✅ Password Service 无状态
- ⚠️ RateLimiter 有状态（需要锁）

## 十二、结论

当前系统性能**中等**，存在多个潜在瓶颈：

1. **数据库连接池配置不合理** - 可能成为高并发瓶颈
2. **限流器单机实现** - 不支持分布式部署
3. **缺少缓存层** - 数据库查询频繁
4. **存在竞态条件** - 并发安全问题

**建议**: 优先优化数据库连接池配置和实现分布式限流，然后逐步添加缓存层和消除竞态条件。
