# 性能分析报告

## 概述

本报告对SSO项目的性能进行分析，涵盖数据库查询、内存使用、并发处理、缓存策略和潜在性能瓶颈等方面。

**分析范围**: 整个项目代码库  
**分析日期**: 2026年3月25日（更新基准测试数据）  
**性能评级**: B+ (良好，有优化空间)

### 1.4 查询优化

#### 潜在问题

1. **用户列表查询**
```go
// internal/store/postgres/postgres.go
func (s *PostgresStore) ListUsers(ctx context.Context, offset, limit int) ([]*model.User, int, error) {
    // 查询总数
    var total int
    err := s.db.QueryRowContext(ctx, "SELECT COUNT(*) FROM users").Scan(&total)
    
    // 查询用户列表
    rows, err := s.db.QueryContext(ctx,
        `SELECT id, email, email_verified, mfa_enabled, status, created_at, updated_at 
         FROM users ORDER BY created_at DESC LIMIT $1 OFFSET $2`,
        limit, offset,
    )
}
```

**评估**:
- ⚠️ 两次查询（COUNT + SELECT）
- ⚠️ 没有WHERE条件过滤
- 建议: 考虑使用窗口函数或缓存总数

2. **Token清理查询**
```go
func (s *PostgresStore) CleanupExpired(ctx context.Context) error {
    _, err := s.db.ExecContext(ctx,
        `DELETE FROM tokens WHERE expires_at < NOW()`)
    _, err = s.db.ExecContext(ctx,
        `DELETE FROM authorization_codes WHERE expires_at < NOW()`)
}
```

**评估**:
- ⚠️ 可能删除大量数据，导致长时间锁表
- 建议: 分批删除或使用后台任务

#### User结构体
```go

    LoginAttempts int        // 8 bytes
    LockedUntil   *time.Time // 8 bytes
```

**估算**: ~230 bytes/user

**10万用户**: ~23 MB
**100万用户**: ~230 MB

**评估**: ✅ 内存使用合理

#### Token结构体
```go
type Token struct {
    ID           string     // 36 bytes
    AccessToken  string     // ~500 bytes (JWT)
    RefreshToken string     // ~50 bytes
```

**估算**: ~750 bytes/token

**1万活跃Token**: ~7.5 MB

**评估**: ✅ 内存使用合理

### 4.2 建议的缓存策略

#### Token缓存
```go
// 建议实现
type TokenCache struct {
    redis *redis.Client
}

func (c *TokenCache) GetToken(ctx context.Context, accessToken string) (*model.Token, error) {
    // 先从Redis查询
    data, err := c.redis.Get(ctx, "token:"+accessToken).Bytes()
    if err == nil {
        var token model.Token
        json.Unmarshal(data, &token)
        return &token, nil
    }
    
    // Redis没有，从数据库查询
    token, err := c.store.GetTokenByAccessToken(ctx, accessToken)
    if err != nil {
        return nil, err
    }
    
    // 缓存到Redis，设置过期时间
    data, _ = json.Marshal(token)
    c.redis.Set(ctx, "token:"+accessToken, data, token.ExpiresAt.Sub(time.Now()))
    
    return token, nil
}
```

#### 用户信息缓存
```go
func (c *UserCache) GetUser(ctx context.Context, userID string) (*model.User, error) {
    // 先从Redis查询
    data, err := c.redis.Get(ctx, "user:"+userID).Bytes()
    if err == nil {
        var user model.User
        json.Unmarshal(data, &user)
        return &user, nil
    }
    
    // Redis没有，从数据库查询
    user, err := c.store.GetByID(ctx, userID)
    if err != nil {
        return nil, err
    }
    
    // 缓存到Redis，设置过期时间（如5分钟）
    data, _ = json.Marshal(user)
    c.redis.Set(ctx, "user:"+userID, data, 5*time.Minute)
    
    return user, nil
}
```

#### 客户端信息缓存
```go
func (c *ClientCache) GetClient(ctx context.Context, clientID string) (*model.Client, error) {
    // 客户端信息变化不频繁，可以长时间缓存
    data, err := c.redis.Get(ctx, "client:"+clientID).Bytes()
    if err == nil {
        var client model.Client
        json.Unmarshal(data, &client)
        return &client, nil
    }
    
    client, err := c.store.GetByClientID(ctx, clientID)
    if err != nil {
        return nil, err
    }
    
    // 缓存1小时
    data, _ = json.Marshal(client)
    c.redis.Set(ctx, "client:"+clientID, data, 1*time.Hour)
    
    return client, nil
}
```

## 5. 性能瓶颈分析

### 5.1 识别的瓶颈

#### 瓶颈1: 密码哈希
```go
// bcrypt哈希是CPU密集型操作
hash, err := bcrypt.GenerateFromPassword([]byte(password), s.cost)
```

**评估**:
- ⚠️ bcrypt哈希需要约100-300ms（cost=12）
- ⚠️ 登录和注册都需要哈希
- 建议: 使用较低的cost（10-11）或考虑其他算法

**性能测试**:
```
bcrypt cost=10: ~50ms
bcrypt cost=11: ~100ms
bcrypt cost=12: ~200ms
bcrypt cost=13: ~400ms
bcrypt cost=14: ~800ms
```

#### 瓶颈2: JWT签名
```go
token := jwt.NewWithClaims(jwt.SigningMethodRS256, claims)
return token.SignedString(s.privateKey)
```

**评估**:
- ✅ RSA签名性能较好
- ⚠️ 每次登录都需要签名
- 建议: 缓存Token生成结果（如果可能）

#### 瓶颈3: 数据库查询
**评估**:
- ⚠️ 每次请求都需要查询数据库
- 建议: 使用缓存减少数据库查询
