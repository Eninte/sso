## 2. 分层架构分析

### 2.1 三层架构

#### Handler 层（表示层）

**职责**:
- HTTP 请求解析
- 参数验证
- 响应格式化
- 错误处理

**特点**:
- ✅ 不包含业务逻辑
- ✅ 依赖 Service 层
- ✅ 使用工具函数（handlerutil）
- ✅ 统一错误响应

**示例**:
```go
func (h *LoginHandler) Handle(w http.ResponseWriter, r *http.Request) {
    // 1. 解析请求
    var req LoginRequest
    if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
        handlerutil.WriteJSONError(w, apperrors.ErrBadRequest)
        return
    }
    
    // 2. 调用 Service
    user, accessToken, refreshToken, err := h.authSvc.Login(r.Context(), req.Email, req.Password)
    if err != nil {
        handlerutil.WriteJSONError(w, err)
        return
    }
    
    // 3. 返回响应
    handlerutil.WriteJSONSuccess(w, map[string]interface{}{
        "user":          user,
        "access_token":  accessToken,
        "refresh_token": refreshToken,
    })
}
```

**评价**: ✅ Handler 层职责清晰，不包含业务逻辑

#### Service 层（业务层）

**职责**:
- 业务逻辑实现
- 事务管理
- 权限控制
- 数据验证

**特点**:
- ✅ 包含核心业务逻辑
- ✅ 依赖 Store 层
- ✅ 使用工具函数（serviceutil, auditutil）
- ✅ 统一错误映射

**示例**:
```go
func (s *AuthService) Login(ctx context.Context, email, password string) (*model.User, string, string, error) {
    // 1. 验证输入
    if err := validator.ValidateEmail(email); err != nil {
        return nil, "", "", err
    }
    
    // 2. 查询用户
    user, err := s.store.GetUserByEmail(ctx, email)
    if err != nil {
        return nil, "", "", serviceutil.HandleStoreError(err, ErrInvalidCredentials)
    }
    
    // 3. 验证密码
    if err := s.passwordSvc.Verify(password, user.PasswordHash); err != nil {
        return nil, "", "", ErrInvalidCredentials
    }
    
    // 4. 生成令牌
    accessToken, err := s.jwtSvc.GenerateAccessToken(user.ID, user.Email, user.Role, []string{})
    // ...
    
    // 5. 记录审计日志
    auditutil.SafeAuditLog(ctx, s.auditSvc, "user_login", user.ID, map[string]interface{}{
        "email": user.Email,
    })
    
    return user, accessToken, refreshToken, nil
}
```

**评价**: ✅ Service 层包含核心业务逻辑，职责明确

#### Store 层（数据层）

**职责**:
- 数据库操作
- 数据持久化
- 查询优化

**特点**:
- ✅ 接口抽象（store.Store）
- ✅ 多实现（postgres, memory, mock）
- ✅ 统一错误定义
- ✅ 事务支持

**示例**:
```go
type Store interface {
    // 用户相关

    // 令牌相关

    GetToken(ctx context.Context, accessToken string) (*model.Token, error)
    RevokeToken(ctx context.Context, accessToken string) error
    
    // ... 其他方法
}
```

**评价**: ✅ Store 层接口抽象良好，支持多实现

### 2.2 依赖关系

```
Handler → Service → Store → Database
   ↓         ↓         ↓
Util     Util      Model
```

**依赖规则**:
- ✅ 单向依赖：上层依赖下层，下层不依赖上层
- ✅ 接口抽象：通过接口解耦
- ✅ 依赖注入：通过构造函数注入依赖
- ✅ 无循环依赖：所有模块无循环引用

**评价**: ✅ 依赖关系清晰，符合分层架构原则

- ✅ 新增 Handler：实现接口即可

- ✅ 新增 Store 实现：实现接口
- ✅ 新增中间件：链式调用

**示例**:
```go

    return &NewHandler{svc: svc}
```
