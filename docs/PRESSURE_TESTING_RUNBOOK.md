# SSO服务 - 压力测试执行手册

本文档是压力测试的执行手册，目标是让测试、后端、运维同学可以直接按步骤准备环境、组织数据、执行场景并记录结果。

现场逐项勾选建议配合 [PRESSURE_TESTING_CHECKLIST.md](./PRESSURE_TESTING_CHECKLIST.md) 使用。

正式沉淀结果时建议配合 [PRESSURE_TESTING_REPORT_TEMPLATE.md](./PRESSURE_TESTING_REPORT_TEMPLATE.md) 使用。

## 1. 适用范围

本手册覆盖以下压力测试范围：

- 公开接口：`/health`、`/.well-known/openid-configuration`、`/.well-known/jwks.json`
- 认证接口：`/api/v1/register`、`/api/v1/login`、`/api/v1/token`、`/api/v1/token/revoke`
- 受保护接口：`/api/v1/userinfo`、`/api/v1/mfa/status`
- OAuth/OIDC：`/api/v1/authorize` → `/api/v1/token` → `/api/v1/userinfo`
- 管理员接口：`/api/v1/admin/health`、`/api/v1/admin/users`、`/api/v1/admin/audit-logs`
- 安全与稳定性：限流、账户锁定、无效Token、突刺、长稳态

## 2. 压测目标

- 找到系统在真实SSO混合流量下的最大稳定吞吐
- 识别登录、注册、userinfo、OAuth流程的容量拐点
- 验证保护机制在高并发和恶意流量下是否按预期生效
- 验证长时间运行时是否存在资源泄漏、连接耗尽、性能退化
- 形成后续版本可复用的容量基线与结果模板

## 3. 环境模式

### 3.1 容量模式

用于测业务极限，不测保护效果。

- 关闭限流
- 关闭或弱化登录保护影响
- 保持数据库、Redis、JWT、审计等真实依赖
- 适合 `login`、`register`、`token`、`userinfo`、OAuth完整流程

### 3.2 保护模式

用于验证系统在恶意流量下是否自我保护。

- 开启限流
- 开启登录失败锁定
- 保留真实认证和存储逻辑
- 适合错误密码风暴、无效Token风暴、重复授权码、同邮箱并发注册

### 3.3 稳态模式

用于识别慢性问题和恢复能力。

- 与容量模式保持一致
- 运行时间为 60-120 分钟
- 适合混合流量、userinfo高频流量、OAuth完整流程

## 4. 工具建议

- 主工具：k6，适合混合场景、阶段控制、阈值和认证链路编排
- 辅助工具：vegeta，适合快速打单接口吞吐基线
- 基准对比：保留 `go test -bench` 用于微基准，不替代真实HTTP压测
- 结果保存：统一存储原始结果、系统指标截图、关键日志和结论摘要

## 5. 执行前检查清单

在开始任何场景之前，逐项确认：

- 服务已经启动，`/health` 返回 200
- 本次压测的目标环境、代码版本、配置版本已经固定
- 已明确当前是容量模式、保护模式还是稳态模式
- 数据库和Redis可用，网络延迟正常
- Metrics 访问方式可用，能够持续拉取指标
- 日志采集、进程监控、主机监控已打开
- 压测结果保存目录已创建
- 压测窗口内不会有其他重任务混入同一环境

## 6. 环境变量检查清单

至少确认以下配置项：

| 配置项 | 用途 | 建议值或要求 |
| ------ | ------ | ------ |
| `SERVER_HOST` | 服务监听地址 | 保持稳定，不在压测中途调整 |
| `SERVER_PORT` | 服务端口 | 与压测目标地址保持一致 |
| `SERVER_ENV` | 运行环境 | 建议使用 development 或专用 perf 环境 |
| `DB_HOST` | 数据库地址 | 指向专用压测库 |
| `REDIS_HOST` | Redis地址 | 指向专用压测实例 |
| `BCRYPT_COST` | 密码哈希成本 | 容量基线需明确记录实际值 |
| `RATE_LIMIT_REQUESTS` | 限流阈值 | 容量模式关闭，保护模式开启 |
| `RATE_LIMIT_WINDOW` | 限流窗口 | 与保护模式报告一起记录 |
| `METRICS_USERNAME` | 指标账号 | 确保可拉取 `/metrics` |
| `METRICS_PASSWORD` | 指标密码 | 确保压测期间不变更 |
| `E2E_ADMIN_EMAIL` | 管理员账号 | 用于管理员接口和后台链路 |
| `E2E_ADMIN_PASSWORD` | 管理员密码 | 确保可稳定登录 |

补充说明：

- `RATE_LIMIT_REQUESTS=0` 适合作为容量模式的显式标记
- `BCRYPT_COST` 必须写入最终报告，否则登录类结果无法横向比较
- 管理员账号需提前确认邮箱已验证、角色为 `admin`

## 7. 数据准备清单

### 7.1 用户池

建议提前准备以下数据池：

| 数据池 | 规模建议 | 用途 |
| ------ | ------ | ------ |
| 普通用户池 | 1000-5000 | 登录、刷新、userinfo、OAuth授权 |
| 普通用户Token池 | 2000-10000 | userinfo、保护链路（非管理员接口） |
| 管理员Token池 | 50-200 | `/api/v1/admin/*` 接口（需admin角色） |
| Refresh Token池 | 1000-5000 | `/api/v1/token` 压测 |
| 注册邮箱池 | 10000+ 唯一邮箱 | `/api/v1/register` 压测 |
| 恶意账号池 | 100-500 | 错误密码、锁定、限流、401/429专项 |
| 管理员账号池 | 5-20 | 生成管理员Token池的源账号 |

### 7.2 数据准备规则

- 所有普通用户都应处于可登录状态
- OAuth用到的用户应已登录或可快速获取登录态
- 注册邮箱必须唯一，避免把 409 当成容量拐点
- `userinfo` 和 OAuth 授权后的读操作应使用普通用户Token池
- 管理员接口（`/api/v1/admin/*`）必须使用管理员Token池，普通用户Token会返回403
- 管理员Token池需从具有 `admin` 角色的账号生成
- 管理员账号不要与普通账号混用
- 恶意账号池要单独标识，避免污染正常成功率统计

## 8. OAuth 客户端准备

建议至少预置两个客户端：

### 8.1 公共客户端

- `client_id`: `public-test-client`
- `public_client`: `true`
- `grant_types`: `authorization_code`, `refresh_token`
- `redirect_uri`: `http://localhost:3000/callback`
- `scopes`: `openid`, `profile`, `email`
- 用途：模拟 SPA 或移动端 PKCE 流程

### 8.2 机密客户端

- `client_id`: `confidential-test-client`
- `public_client`: `false`
- `grant_types`: `authorization_code`, `refresh_token`
- `redirect_uri`: `http://localhost:3000/callback`
- `scopes`: `openid`, `profile`, `email`
- 用途：模拟服务端应用授权码换 Token

### 8.3 OAuth 数据准备检查

- 两个客户端都已成功写入数据库
- `redirect_uri` 已验证可用
- 机密客户端的 `client_secret` 已记录在压测凭据中
- OAuth场景所用测试用户能够完成授权链路
- PKCE 场景具备 `code_verifier` 和 `code_challenge` 生成能力

## 9. 场景执行顺序

推荐按以下顺序执行，避免一开始就把变量混在一起：

1. S1 公开读接口基线
2. S2 登录单接口
3. S3 注册单接口
4. S4 Refresh Token 单接口
5. S5 UserInfo 高频读取
6. S6 OAuth 公共客户端完整流程
7. S7 OAuth 机密客户端完整流程
8. S8 混合流量
9. S9 安全保护专项
10. S10 突刺与恢复
11. Soak Test

## 10. 统一负载阶段

每个核心场景都建议按相同节奏推进：

| 阶段 | 目标 | 负载建议 | 时长建议 |
| ------ | ------ | ------ | ------ |
| 冒烟 | 检查脚本与接口是否正确 | 1-5 并发 | 1-2 分钟 |
| 基线 | 获取轻负载延迟基准 | 10 / 20 / 50 并发 | 每档 2 分钟 |
| 阶梯加压 | 找系统拐点 | 50 → 100 → 200 → 300 → 500 → 800 | 每档 3-5 分钟 |
| 峰值保持 | 验证接近极限时的稳定性 | 取拐点前一档 | 10-15 分钟 |
| 突刺 | 验证峰值冲击与恢复 | 1 倍 → 3-5 倍 → 1 倍 | 1-3 分钟突刺 |
| 稳态 | 验证资源曲线是否平台化 | 高峰 50%-70% | 60-120 分钟 |

## 11. 场景执行清单

### S1 公开读接口基线

目标：

- 建立服务最轻路径吞吐基线
- 确认 HTTP 层、路由层、网络层没有明显瓶颈

接口：

- `GET /health`
- `GET /.well-known/openid-configuration`
- `GET /.well-known/jwks.json`

执行要点：

- 先压 `/health` 建立最低成本延迟基线
- 再压两个 `.well-known` 端点，验证公开元数据能力
- 若这一步都不稳定，后续业务压测结论无参考意义

建议负载：

- 基线：50 / 200 / 500 RPS
- 阶梯：1000 / 1500 / 2000 RPS
- 峰值保持：拐点前一档持续 10 分钟

验收：

- 5xx 接近 0
- P95 稳定
- CPU 与内存曲线平滑

### S2 登录单接口

目标：

- 识别 bcrypt、数据库查询、JWT签名的综合瓶颈

接口：

- `POST /api/v1/login`

数据准备：

- 1000-5000 个已验证普通账号
- 每轮随机分布请求，避免单账号过热

执行要点：

- 记录实际 `BCRYPT_COST`
- 统计 200、401、429、5xx 分布
- 若开启保护模式，单独记录锁定和限流比例

建议负载：

- 并发：10 / 20 / 50 / 100 / 200 / 300
- 稳态：取拐点前一档跑 30 分钟

验收：

- 容量模式下 5xx < 1%
- 明确登录的稳定吞吐和 CPU 拐点

### S3 注册单接口

目标：

- 识别写路径容量、唯一约束、审计链路影响

接口：

- `POST /api/v1/register`

数据准备：

- 10000+ 唯一邮箱
- 固定强密码

执行要点：

- 只统计唯一邮箱成功注册的容量
- 同邮箱并发注册作为保护模式单独测试
- 若邮件链路不可控，应记录是否接入测试SMTP或黑洞服务

建议负载：

- 并发：5 / 10 / 20 / 50 / 100
- 峰值保持：拐点前一档持续 10 分钟

验收：

- 成功率稳定
- 非预期 5xx 可定位
- 409 仅出现在故意构造的冲突场景

### S4 Refresh Token 单接口

目标：

- 验证会话续期的稳定吞吐和延迟

接口：

- `POST /api/v1/token`

数据准备：

- 1000-5000 个 refresh token

执行要点：

- 避免重复使用同一 refresh token 导致业务失败污染统计
- 区分容量压测和非法 token 压测

建议负载：

- 并发：20 / 50 / 100 / 200 / 300 / 500

验收：

- 刷新成功率稳定
- 非法 token 场景返回 4xx，不应大面积触发 5xx

### S5 UserInfo 高频读取

目标：

- 验证最常见受保护读路径的真实吞吐

接口：

- `GET /api/v1/userinfo`

数据准备：

- 2000-10000 个 access token

执行要点：

- 不要在场景内临时登录获取 token
- 重点观察 P95、P99、缓存命中、认证链路耗时

建议负载：

- 并发：100 / 300 / 500 / 800 / 1000
- 稳态：高峰 60% 跑 60 分钟

验收：

- 成功率接近 100%
- P95 稳定，无持续劣化

### S6 OAuth 公共客户端完整流程

目标：

- 模拟 SPA/移动端 PKCE 完整授权码流程

链路：

- `GET /api/v1/authorize`
- `POST /api/v1/token`
- `GET /api/v1/userinfo`

数据准备：

- 公共客户端
- 已登录用户池
- PKCE 验证参数生成能力

执行要点：

- 验证 state、code、code_verifier 的完整传递
- 统计授权成功率、换 Token 成功率、userinfo 成功率
- 错误码需区分业务 4xx 和系统 5xx

建议负载：

- 并发：10 / 20 / 50 / 100 / 150

验收：

- 授权与换 Token 成功率稳定
- 非法 PKCE 请求只应产生 4xx

### S7 OAuth 机密客户端完整流程

目标：

- 模拟服务端应用授权码换 Token

链路：

- `GET /api/v1/authorize`
- `POST /api/v1/token`
- `GET /api/v1/userinfo`

数据准备：

- 机密客户端
- `client_secret`
- 已登录用户池

执行要点：

- 单独统计 secret 校验失败场景
- 记录 code 重用时的响应分布

建议负载：

- 并发：10 / 20 / 50 / 100 / 200

验收：

- 正常链路成功率稳定
- secret 错误和 code 重用只返回 4xx

### S8 混合流量

目标：

- 用接近真实 SSO 业务画像的流量验证整体系统稳定性

推荐流量配比：

- 40% `GET /api/v1/userinfo`
- 20% `POST /api/v1/token`
- 15% `POST /api/v1/login`
- 8% `POST /api/v1/register`
- 10% OAuth 公共客户端完整流程
- 5% OAuth 机密客户端完整流程
- 2% 管理员接口

执行要点：

- 记录整体吞吐和各子场景成功率
- 观察数据库与Redis是否出现排队放大
- 重点看是否某个子场景拖垮整体系统

建议负载：

- 先以单接口结论倒推出混合总负载
- 再从 25% / 50% / 75% / 100% 目标生产峰值逐档推进

验收：

- 5xx < 1%
- 主链路成功率稳定
- 没有单点场景拖垮整体

### S9 安全保护专项

目标：

- 验证系统在恶意流量下自我保护而不雪崩

子场景：

- 错误密码高频登录
- 同邮箱并发注册
- 无效 Token 风暴
- 无效授权码与重复 code
- 错误 `client_secret`
- 错误 `code_verifier`

执行要点：

- 必须在保护模式下执行
- 单独统计 401、409、429 比例
- 观察保护触发后主服务是否仍能处理正常请求

验收：

- 429、401、409 行为符合预期
- 不出现大面积 5xx

### S10 突刺与恢复

目标：

- 验证统一登录高峰、活动峰值、客户端批量刷新时的冲击承受能力

执行方式：

- 先以混合流量跑常态
- 瞬时拉升到 3-5 倍
- 观察 1-3 分钟后回落到常态

验收：

- 峰值期间允许有限退化
- 峰值结束后 P95 和错误率可回归到接近基线

### Soak Test

目标：

- 识别慢性泄漏、依赖抖动、长期性能退化

建议：

- 以 S8 混合流量的 50%-70% 强度持续 60-120 分钟
- 每 5-10 分钟截取一次系统资源和业务指标

验收：

- 内存、goroutine、连接池进入平台期
- 没有持续性错误率上升

## 12. 观测指标

### 12.1 压测工具输出

- 实际吞吐
- 平均延迟
- P90、P95、P99
- 最大延迟
- 成功率
- 4xx、5xx 分布

### 12.2 应用指标

- `http_requests_total`
- `auth_login_total`
- `auth_login_failed_total`
- `auth_token_refresh_total`
- `security_rate_limit_total`
- `security_invalid_token_total`
- `cache_hits_total`
- `cache_misses_total`

### 12.3 主机与依赖指标

- CPU
- RSS 内存
- goroutine 数
- GC 次数与 pause
- 数据库连接数与慢查询
- Redis RTT、错误数、连接数

## 14. 执行后复盘清单

场景结束后，统一检查：

- 是否保存了原始压测结果
- 是否保存了关键监控截图
- 是否记录了核心配置项

- 是否区分了业务失败和系统失败
- 是否输出了可执行的优化建议
- 是否将本次结果加入版本对比基线

## 15. 注意事项

- 登录与注册结果会显著受 `BCRYPT_COST` 影响，必须记录实际值
- 容量模式和保护模式不能混跑，否则结论不可比较
- `userinfo` 压测必须使用预生成Token池，不能在场景内顺便登录
- OAuth 压测必须准备合法客户端和合法 `redirect_uri`
- 注册压测必须使用唯一邮箱
- 保护模式的目标是验证 401、409、429 是否正确，不是追求最高吞吐
- 当前应用内 HTTP 耗时指标不能直接代替压测工具分位数结果
